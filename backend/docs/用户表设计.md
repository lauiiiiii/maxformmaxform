## 用户表设计（与当前代码一致）

- 所属集合：`users`
- 代码来源：`backend/src/models/User.js`（通过 `database.getModel('User')` 统一获取）
- 存储引擎：MongoDB（通过数据库适配器接入 Mongoose）
- 时间戳：开启 `timestamps: true`，自动维护 `createdAt`、`updatedAt`

### 数据库接入方式

- 项目已切换为“数据库适配器”模式，`DB_CLIENT` 环境变量用于选择底层数据库实现。
- 当前默认值为 `mongodb`，因此本表依旧由 MongoDB/Mongoose 提供；路由与脚本通过 `database.getModel('User')` 获取模型。后续扩展其它数据库时，仅需新增适配器并调整 `DB_CLIENT`。
- 入口文件（`app.cjs` / `app.js`）和脚本（如 `initAdmin.js`）都会统一调用适配器的 `connect`、`disconnect`、`isConnected`，健康检查 `/health` 会返回 `dbClient`、`dbOk`、`mongoOk` 字段以便观测状态。

### 双数据库架构定位（本表在 MongoDB）

- 我们的双库分工：
  - 事务/主数据（用户、角色、部门、问卷定义等）→ MongoDB（权威库）
  - 大体量行为/明细用于统计分析（答卷明细、统计快照等）→ ClickHouse（分析库）
- 本文档的 `users` 表仅存放在 MongoDB 中，不复制到 ClickHouse。
- 如需做用户维度的 BI，可将“登录/操作审计事件”异步写入 ClickHouse（而非复制整个用户表）。

可选：登录事件审计表（ClickHouse，尚未实现，仅作建议）

```sql
-- 库：analytics（示例）
-- 表：login_events（示例）
CREATE TABLE IF NOT EXISTS analytics.login_events (

  ip          String          COMMENT '来源 IP',
  ua          String          COMMENT 'User-Agent（可选）'
) ENGINE = MergeTree
ORDER BY (ts, user_id);
```

### 字段定义

| 字段名        | 类型       | 必填 | 默认值   | 约束/说明 |
| ------------- | ---------- | ---- | -------- | --------- |
| _id           | ObjectId   | 是   | 自动生成 | 主键 |
| username      | String     | 是   | -        | 唯一；去首尾空格；长度 3-30 |
| email         | String     | 是   | -        | 唯一；强制小写；正则格式校验 |
| password      | String     | 是   | -        | 最小长度 6；入库前使用 bcrypt 哈希 |
| role          | String     | 否   | user     | 枚举：`admin`、`user` |
| avatar        | String     | 否   | 空字符串 | 头像 URL（可选） |
| isActive      | Boolean    | 否   | true     | 账户是否启用（逻辑禁用可用此字段） |
| lastLoginAt   | Date       | 否   | -        | 最近一次登录时间（登录成功时更新） |
| createdAt     | Date       | -    | now      | 由 Mongoose 维护 |
| updatedAt     | Date       | -    | now      | 由 Mongoose 维护 |

说明：`unique: true` 将在 MongoDB 中创建唯一索引（首次连接或 `syncIndexes` 时生效）。

### 索引

- 唯一索引：`username`、`email`
- 如需额外查询优化，可按需增加：例如 `{ role: 1, isActive: 1 }`

### 校验与序列化行为

- 密码哈希：`pre('save')` 钩子对 `password` 进行 `bcrypt` 哈希（salt rounds 10）。
- 比对方法：`comparePassword(candidate)` 用于登录时校验明文密码。
- 安全序列化：`toJSON()` 会剔除 `password` 字段，避免通过接口泄露。

### 与接口的对应关系（当前实现）

- 注册：`POST /api/auth/register`
  - 请求体：`{ username, email, password }`（三者必填，且 `username`/`email` 全局唯一）
  - 行为：创建用户（`role` 默认为 `user`），返回 `token` 与 `user`（无 `password`）。
- 登录：`POST /api/auth/login`
  - 请求体：`{ username, password }`
  - 行为：校验密码，更新 `lastLoginAt`，返回 `token` 与 `user`。
- 获取当前用户：`GET /api/auth/me`
  - 请求头：`Authorization: Bearer <JWT>`
  - 行为：校验令牌，返回当前用户信息。
- 用户管理（简单 CRUD，用于后台/脚本）：`/api/users`
  - `GET /api/users` 列表；`GET /api/users/:id` 详情；`POST /api/users` 创建；`PUT /api/users/:id` 更新；`DELETE /api/users/:id` 删除。

### 认证与令牌

- 令牌类型：JWT，签名密钥 `JWT_SECRET`（默认 `dev-secret`），有效期 `JWT_EXPIRES_IN`（默认 `7d`）。
- 令牌存储：不入库；登录/注册时下发给客户端，由前端存储（localStorage/cookie）。

### 状态与禁用

- `isActive` 可用于逻辑禁用账号；如需强制在登录环节拦截，可在登录逻辑中增加 `isActive` 判断（当前代码未拦截，仅提供字段）。

### 与角色/权限/部门的关系（当前代码状态）

- 用户的 `role` 为简单字符串枚举（`admin`/`user`）。
- `Role`、`Permission`、`Account`、`Dept` 等模型已定义雏形，但未在用户鉴权流中启用；后续若需要更细粒度 RBAC，可：
  - 将 `role` 替换为 `roles: [ObjectId]` 关联 `roles` 集合；
  - 引入 `permissions` 并在路由中做权限校验；
  - 在 `User` 中新增 `deptId` 以支持组织结构管理。

### 扩展建议（尚未实现，按需演进）

- 企业多租户：为 `User` 增加 `tenantId/enterpriseId` 字段，接口按租户隔离。
- 审计与风控：登录失败计数、密码修改日志、敏感操作审计等。
- 安全加强：密码复杂度策略、两步验证（2FA）、登录地异常告警。
- 索引治理：对高频查询场景（如按邮箱/用户名模糊搜索）增加合适索引并评估覆盖率。
