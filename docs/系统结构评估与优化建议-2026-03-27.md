# 系统结构评估与优化建议
更新时间：2026-03-29

## 1. 评估范围

本次评估基于当前仓库源码、2026-03-29 当天已通过的自动化验证结果，以及最近完成的流程/题库管理后端落地，不再沿用旧版 MongoDB / ClickHouse 双库叙述。

重点检查范围：

- `backend/` 当前真实启动入口、分层结构，以及问卷域和管理域的服务化程度
- `frontend/` 当前页面层、API 层和编辑器 composable 的边界清晰度
- `shared/` 是否持续承担跨前后端 DTO、错误码和结构归一化职责
- 自动化验证是否能够覆盖当前主链路，而不是仅覆盖历史实现

## 2. 本次结论

当前系统已经进入“可持续迭代的模块化单体”阶段，核心矛盾不再是是否重写，而是如何把已经验证有效的 `route + policy + service + repository + contract` 模式继续推广到更多模块，并同步修正文档与回归覆盖。

截至 2026-03-29，更准确的判断是：

- 后端主线已经稳定在 `Express + Knex + MySQL`
- 问卷、答卷、文件链路已经形成较完整的服务化分层
- 管理域不再只覆盖部门、角色、职位、消息、审计，流程和题库仓库也已进入正式后端路由
- 前端 `frontend/src/api/flows.ts`、`frontend/src/api/repos.ts` 已切回真实 HTTP 接口，不再是本地占位实现
- `shared/management.contract.js` 已经真正成为管理域 DTO 的共享来源，而不只是类型声明集合
- 当前最需要修正的不是结构方向，而是旧文档和旧结论仍把流程/题库描述成“未落地能力”

一句话结论：

**当前重点不是推翻架构，而是继续扩大服务化覆盖、统一共享 contract，并及时清理已经失真的系统说明文档。**

## 4. 已确认的结构进展

### 4.1 后端服务化范围继续扩大

和前一轮评估相比，最近最关键的变化不是文件数量，而是服务边界继续清晰：

- 问卷链路已稳定分离为查询、写入、上传、结果统计
- 答卷链路已稳定分离为查询、批量删除、导出
- 文件链路已进入 `query/command` 分离
- 管理域在部门、角色、职位、文件夹、消息、审计之外，新增流程和题库正式服务链路

这说明当前后端已经不是“问卷域服务化、管理域大量直连 model”的混合早期状态，而是在继续向统一风格收口。

### 4.2 结果统计进入“实时聚合 + 快照缓存”阶段

`backend/src/services/surveyResultsService.js` 当前已经引入 `SurveyResultsSnapshot`，说明：

- 系统仍保留 JSON 写模型带来的题型迭代灵活性
- 同时开始使用轻量快照降低重复统计成本
- 当前还不是独立读模型架构，但也已经明显脱离“每次请求都全量重算”的最初阶段

### 4.3 管理域共享 DTO 进入真实运行阶段

本轮相比之前最大的结构性补强点是：

- 流程和题库不再只是共享 DTO
- 其后端路由、服务、仓储、模型和迁移已经全部落地
- 前端 API 也已切回真实接口
- 管理域 `shared contract -> frontend api -> backend route -> service -> repository -> model` 链路已经对齐

这使得“管理域共享 contract 化”不再是规划，而是已进入运行状态。

## 5. 当前仍存在的主要问题

### P1. 管理域分层风格仍未完全统一

虽然流程和题库已经补齐，但管理域仍存在风格差异：

- 有的模块只要求登录即可访问列表
- 有的模块要求管理员访问
- 访问控制策略的粒度还没有形成统一准则
- 审计、消息、事务边界在不同模块中的使用强度不一致

这不是阻塞问题，但会持续拉低整体一致性和后续接手成本。

### P1. 浏览器级回归仍偏重问卷主链路

当前浏览器验证已覆盖问卷、答卷、上传等高价值路径，但流程和题库刚落地，尚未看到对应的浏览器级管理回归沉淀。

缺口主要在：

- 流程管理页面真实接口联调回归
- 题库管理页面创建、删除、抽屉加载、题目录入回归
- 管理域权限差异场景回归

### P1. 文档系统存在明显滞后

当前仓库仍有多份文档写着：

- 流程/题库仍是占位 API
- 后端尚无正式路由
- 前端仍依赖假实现

这些信息现在已经失真，会直接误导后续开发与评估判断。

### P2. 共享 contract 仍以 DTO 和归一化为主

当前 `shared/` 已覆盖大量 DTO 和结构归一化，但仍未形成更系统的：

- 管理域统一错误码族
- 更完整的管理域查询参数 contract
- 跨模块共享的校验助手

### P2. 流程模块能力仍偏基础

流程正式路由已经落地，但当前仍属于基础 CRUD 阶段：

- 只有列表、创建、更新、删除
- 尚无流程步骤定义、审批节点、绑定业务对象等更复杂语义
- 目前更像“流程目录/模板入口”，还不是完整工作流引擎

这不是缺陷，而是需要在文档中准确表述，不要把它误写成成熟 BPM 能力。

## 6. 2026-03-29 实测结论

本轮不是纯静态阅读，已经完成以下验证：

- `backend/` 下执行 `npm.cmd test`，通过 `75 / 75`
- `backend/` 下单跑 `node --test test/management.routes.test.js`，流程/题库新增路由测试通过
- `frontend/` 下执行 `npm.cmd run build`，通过

本轮新增验证覆盖的关键点包括：

- `GET /api/flows` 列表返回共享 `FlowDTO`
- `POST /api/flows` 对非法状态值进行校验
- `GET /api/repos` 返回共享题库仓库 DTO
- `POST /api/repos/:id/questions` 会先校验 repo 存在
- `DELETE /api/repos/:id/questions/:questionId` 走正式 service 流程删除

当前可以明确确认：

- 流程与题库不再是“文档层规划能力”
- 已经进入可联调、可测试、可构建验证的真实后端能力

## 7. 下一步建议

第一优先级：

- 清理仍将流程/题库写成占位能力的文档
- 为流程管理和题库管理补浏览器级 E2E
- 统一管理域访问控制准则，明确哪些模块是“登录可见”，哪些是“仅管理员”

第二优先级：

- 继续把管理域查询参数、错误码和分页结构沉淀到共享 contract
- 让审计、消息、事务边界在更多管理模块中保持一致
- 在 `README`、`开发指南`、`API接口说明` 中同步新路由说明

第三优先级：

- 如果流程模块继续扩展，再决定是否引入流程步骤、节点配置和审批执行模型
- 如果题库模块继续扩展，再补编辑、批量导入、分类标签和题目复用能力

## 8. 统一口径

当前系统应统一描述为：

- 后端是 `Express + Knex + MySQL` 的模块化单体
- 问卷、答卷、文件链路已形成较稳定的服务化分层
- 管理域已覆盖用户、角色、部门、职位、文件夹、消息、审计、流程、题库仓库
- 流程和题库已经具备正式后端路由与前端真实 API，不再属于占位实现
- `shared/` 已成为题型语义、问卷 contract、管理域 DTO 和分页结构的统一事实源
- 当前主要任务是继续扩展分层覆盖、统一管理域规范、补齐浏览器回归，并清理滞后文档
