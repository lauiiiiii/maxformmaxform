# 企业级问卷系统部署与功能规范（对齐当前实现）

> 2026-03-28 注：本文档仍包含旧版双库架构与适配器方案描述，当前仓库实际实现已切换为 `Express + Knex + MySQL` 主线。部署判断请先以 `README0326.md`、`backend/README.md` 与 `docs/系统结构评估与优化建议-2026-03-27.md` 为准。

本文档结合现有代码与部署方式，给出可落地的企业级部署与功能规范，适用于私有化与内网环境。

## 项目定位

企业级独立部署的问卷/表单平台，支持多企业（租户）使用，满足权限分级、数据隔离与合规需求。

## 架构说明（适配器 + 双数据库）

- 前端：Vue 3 + TypeScript + Vite（静态资源可由 Nginx/网关托管）
- 后端：Node.js + Express，所有数据访问经 `src/database` 适配器门面（默认入口 `backend/app.cjs`，端口 63002；`simple-server.js` 为单库演示版，不经过适配层）
- 数据库：
  - 主库通过适配器可插拔，默认实现为 MongoDB（需启用副本集以支持 Change Streams）；亦可扩展 MySQL、PostgreSQL、SQL Server、SQLite、人大金仓、达梦、OceanBase 等驱动
  - ClickHouse（分析库）通过 HTTP 接口写入/查询，存放答题明细宽表 `trustform.survey_answers`
- 复制链路（默认）：提交答卷写入 Mongo → 后台复制器基于 Mongo Change Streams 异步写入 ClickHouse
- 健康检查：`GET /health` 返回 `dbClient`、`dbOk`、`mongoOk`、`clickhouseOk` 等字段，方便探针识别当前适配器
- 分析接口：`GET /api/surveys/:id/analytics` 直接查询 ClickHouse 聚合结果

参考：详细开发与环境配置见 `docs/开发指南.md`、`docs/环境安装指南.md`。

## 数据库适配策略

- 所有业务读写通过 `backend/src/database` 门面获取模型或仓库，避免控制器直接依赖某个驱动；生产环境部署需保持该目录与适配器实现一致。
- 默认 `DB_CLIENT=mongo` 使用 Mongoose 模型注册；如需切换数据库，实现对应 `<Engine>Adapter` 并在 `database/index.js` 注册即可。
- 健康检查返回的 `dbClient`、`dbOk` 可用于探针与监控，变更适配器后无需调整探针脚本。
- `backend/simple-server.js` 仅适用于单机演示，不包含多数据库适配逻辑及 ClickHouse 复制链路，企业部署请使用完整版 `app.cjs`。
- 由于复制器依赖 Mongo Change Streams，若主库迁移至非 Mongo 适配器，需同步提供替换的数据捕获与写入 ClickHouse 方案。

## 企业级功能清单

1. 企业与成员管理
   - 多企业账号，企业管理员/成员分级
   - 用户邀请、成员管理、权限分配（RBAC）
2. 问卷管理
   - 问卷创建、编辑、复制、删除、分组、标签
   - 模板库与题库管理
3. 题型系统
   - 后端权威题型为字符串枚举，前端内部存在 legacy 数字枚举并做双向映射（见 `docs/题型规范.md`）
4. 问卷发布与收集
  - 多渠道发布（链接、二维码、邮件）
  - 回收设置（截止时间、次数/IP 限制、去重）
  - 实时数据收集（默认落地 Mongo 主库；如更换适配器则写入对应数据库），异步分析（ClickHouse）
5. 数据分析与报告
   - ClickHouse 聚合分析接口，前端可视化（ECharts 等）
   - 数据导出（Excel 已实现，PDF 可选）
   - 企业/角色数据隔离
6. 审批与流程（可选）
   - 发布审批、数据导出审批，流程日志
7. 安全与合规
   - HTTPS、CORS 白名单、鉴权与审计
   - 数据备份与留存策略、敏感字段加密（可选）
8. 系统与运维
   - 健康检查、监控告警、操作日志、容量规划

## 多租户与权限分级

- 强制企业（tenant）隔离：所有核心数据包含 `enterpriseId`，查询必须带入该条件
- 角色模型（示例）：企业管理员、部门管理员、问卷编辑者、只读分析者
- 前端路由与菜单基于角色动态授权；后端统一基于 JWT 校验与 RBAC 控制
- 重要操作需审计（创建/发布/导出/删除），支持按企业与用户检索

## 安全与合规规范

- 身份认证与授权
  - JWT（建议设置合理过期时间与刷新机制），后端验证签名与权限；必要时支持单点登录接入
  - CORS 仅放行受信来源（`FRONTEND_URL`）
- 数据安全
  - 传输 HTTPS；敏感字段（如手机号/邮箱）可按需加密或脱敏存储
  - 最小权限访问数据库账号；区分只读/写入账号（ClickHouse 可使用独立分析用户）
- 接口安全
  - 速率限制、防重放、防注入（ClickHouse 查询使用受控 SQL 拼接并做参数校验）
  - 管理类接口可叠加 IP 白名单/二次校验
- 日志与审计
  - 应用层操作日志与访问日志集中收集（如 ELK/云日志），保留期按合规要求设置
  - 定期审计角色与权限变更

## 部署形态与网络

- 最小生产形态（推荐）
  - 1 台应用节点（Node.js 18+，PM2 进程守护）
  - MongoDB（单节点副本集即可支持 Change Streams；生产可用 3 节点，或根据适配器替换为其他主库方案）
  - ClickHouse（单节点；大规模可水平扩展至集群）
  - Nginx 反向代理与静态托管
- 端口与网段
  - 后端：63002（`app.cjs`）
  - 前端：对外 80/443；开发态 63000
  - ClickHouse：8123（HTTP）
  - 数据库与内部端口建议置于内网网段，只通过网关暴露

## 配置与环境变量（关键项）

- `PORT=63002`
- `DB_CLIENT=mongo`（指定主库适配器，例如 `mongo`、`mysql`、`postgres`）
- `MONGODB_URI`（Mongo 适配器需要，建议副本集：`mongodb://host:27017/db?replicaSet=rs0`）
- 其他数据库连接串（示例：`MYSQL_URI`、`PG_URI` 等，根据适配器定义）
- `CLICKHOUSE_URL`（如 `http://host:8123`）
- `CLICKHOUSE_DB`、`CLICKHOUSE_USER`、`CLICKHOUSE_PASSWORD`
- `JWT_SECRET`（定期轮换）
- `FRONTEND_URL`（CORS 白名单）

## 部署与运维建议

- 进程与代理
  - 后端使用 PM2 启动 `app.cjs`，Nginx 代理 `/api` → `http://backend:63002`
  - 前端构建产物由 Nginx 静态托管（`try_files ... /index.html`）
- 健康检查与就绪
  - `GET /health` 应显示 `dbClient`、`dbOk`、`mongoOk`、`clickhouseOk` 等字段，其中关键项为 `dbOk: true` 与对应子项为 `true` 后方可放流量
  - Mongo 副本集需先 `rs.initiate()`；ClickHouse 启动并可达 8123
- 配置核查
  - 部署前确认 `.env` 中的 `DB_CLIENT` 与对应连接串已按目标数据库适配器配置，并确保 `backend/src/models/index.js` 对应模型注册与适配器实现兼容
- 监控与日志
  - PM2 指标与日志、Nginx 访问/错误日志、应用操作日志集中采集
  - 关键接口错误率、响应时间、资源使用（CPU/内存/磁盘/连接数）需告警
- 备份与恢复（建议基线）
  - Mongo：每日 `mongodump`，保留 7–30 天；每月演练恢复
  - ClickHouse：使用 BACKUP/RESTORE 或卷快照，确保与 Mongo 备份点时间接近
  - 备份包含环境与版本元数据，便于跨环境恢复
- 容量与性能
  - Mongo 建议为高频查询字段建索引（见 `开发指南.md` 索引段落）
  - ClickHouse 表 `MergeTree`，按 `projectId, questionId, createdAt` 排序；可配置数据留存与归档
  - 大批量导入建议走批处理（JSONEachRow 批量提交）

## 数据同步与回灌

- 在线复制：Mongo → ClickHouse 由后台复制器（Change Streams）异步写入
- 如切换到非 Mongo 主库，需要重新设计数据捕获流程（如 Debezium、CDC 日志解析等）以维持 ClickHouse 同步能力
- 回灌脚本：历史答卷可离线遍历 `Answer` 集合写入 ClickHouse（一次性/按需运行）
- 失败重试：建议增加批量缓冲与失败重试队列（生产可按需增强）

## 接口与页面（简要对齐）

- 回答提交：`POST /api/surveys/:id/responses`（默认写入 Mongo 主库；复制器异步写入 ClickHouse）
- 分析查询：`GET /api/surveys/:id/analytics`（查 CH）
- 健康检查：`GET /health`
- 登录与角色：JWT 鉴权，路由按角色授权
