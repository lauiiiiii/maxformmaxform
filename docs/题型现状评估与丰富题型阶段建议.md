# 题型现状评估与丰富题型阶段建议

基于源码状态更新时间：2026-03-26

本文仅以当前仓库实际代码为准，评估题型是否已经形成“创建、保存、预览、填写、提交、结果统计”的闭环，不再沿用 2026-03-25 之前的旧判断。

## 1. 本次复核结论

截至 2026-03-26，项目的题型能力已经从“中等题型可用”进一步推进到“扩展矩阵与比重题也具备真实闭环”，与旧文档相比需要做出以下修正：

- 编辑器、填写页、提交校验、结果统计已经覆盖：
  - `input`
  - `textarea`
  - `radio`
  - `checkbox`
  - `dropdown(uiType=7)`
  - `slider`
  - `ranking`
  - `upload`
  - `date`
  - `matrix(uiType=20/21/24)`
  - `rating(uiType=29)`
  - `scale(uiType=30/34)`
  - `ratio(uiType=36)`
- `stage_explain(uiType=18)` 作为说明型内容块已形成“创建、保存、渲染”闭环，但不属于作答题型。
- `/api/surveys/:id/results` 的题目级统计已被抽到共享模型 `shared/questionModel.js`，后端与前端展示口径不再各写一套。
- 前端答卷分析面板已经不再是“题目统计真、图表假”的状态：
  - 题目图表通过 `QuestionAnalyticsChart.vue` 按真实数据渲染
  - 收集趋势图、地域分布、系统环境图已消费真实接口数据
  - Excel、附件包、SPSS 语法导出动作都已形成可执行按钮与反馈

新的判断应当是：

**项目已进入“题型收口、分析体验增强与浏览器级回归补齐阶段”，而不是“矩阵多选 / 矩阵下拉 / 比重题尚未交付”的阶段。**

## 2. 当前真实题型能力边界

### 2.1 单一事实源

题型的共享注册表位于：

- `shared/questionTypeRegistry.js`

前端和后端分别通过以下文件复用：

- `frontend/src/utils/questionTypeRegistry.ts`
- `backend/src/constants/questionTypes.js`
- `backend/src/utils/questionSchema.js`

这意味着：

- legacy `uiType`
- server `type`
- 是否实现
- 是否需要选项
- 默认选项
- 特殊渲染类型
- 提交校验种类

已经开始集中收口，不应再在各页面各写一套独立枚举。

### 2.2 当前共享注册表中已实现的 legacy 题型

`shared/questionTypeRegistry.js` 中 `implemented=true` 的题型为：

| legacy uiType | server type | 说明 |
|---|---|---|
| `1` | `input` | 填空题 |
| `2` | `textarea` | 简答题 |
| `3` | `radio` | 单选题 |
| `4` | `checkbox` | 多选题 |
| `7` | `radio` | 下拉题，渲染时按 `dropdown` 处理 |
| `8` | `slider` | 滑动条题 |
| `11` | `ranking` | 排序题 |
| `13` | `upload` | 文件上传 |
| `14` | `date` | 日期题 |
| `18` | `input` | 段落说明，渲染为 `stage_explain` |
| `20` | `matrix` | 矩阵单选 |
| `21` | `matrix` | 矩阵多选 |
| `24` | `matrix` | 矩阵下拉 |

未实现但仍保留入口/枚举的主要包括：

- `31` 评分单选
- `32` 评分多选

## 3. 各环节闭环评估

### 3.1 编辑器

编辑器主入口位于：

- `frontend/src/views/survey/CreateSurveyPage.vue`

当前已确认：

- 基础题型和扩展题型均可创建并保存
- `matrix` 支持行维度编辑，以及 `single / multiple` 选择模式
- `matrix dropdown` 通过 `uiType=24` 和共享渲染类型驱动
- `ratio` 已纳入选项题型分支，可直接配置选项并落库
- `rating` 支持评分范围配置
- `scale` 支持量表范围与标签配置
- `upload` 支持数量、大小、后缀限制配置

需要注意：

- 编辑器内部仍有较多 legacy 编码判断，维护成本偏高
- 题型注册表已建立，但部分页面逻辑仍未完全收口到共享配置

### 3.2 预览与填写页

填写页位于：

- `frontend/src/views/survey/FillSurveyPage.vue`
- `frontend/src/views/survey/FillSurveyMobilePage.vue`

当前已存在真实渲染分支的类型：

- `input`
- `textarea`
- `radio`
- `checkbox`
- `dropdown`
- `date`
- `slider`
- `ranking`
- `upload`
- `rating`
- `scale`
- `matrix single`
- `matrix multiple`
- `matrix dropdown`
- `ratio`
- `stage_explain`

补充说明：

- 移动端当前复用 PC 填写逻辑，因此题型支持范围一致。
- `ratio` 会在前端实时提示当前总占比，并在未满 `100%` 时给出明显错误态。
- `upload` 走“先上传文件，再提交答卷引用”的模式，不是直接把二进制塞进答卷。

### 3.3 提交校验

提交流程关键逻辑位于：

- `backend/src/utils/questionSchema.js`
- `backend/src/routes/surveys.js`

当前校验能力已经覆盖：

- 发布前标题/题目/选项结构校验
- `radio / checkbox / ranking` 选项合法性校验
- `slider / rating / scale` 数值校验
- `matrix single` 行列结构与单值作答校验
- `matrix multiple` 行列结构与数组作答校验
- `ratio` 百分比数字、取值范围与合计 `100` 校验
- `upload` 文件数量、大小、类型校验
- 必填校验
- 截止时间校验
- 重复提交限制

上传题额外校验：

- 文件是否属于当前问卷
- 文件是否属于当前题号
- `submissionToken` 是否一致
- `uploadToken` 是否匹配

这意味着项目已经不再处于“答卷原样写入 JSON”的阶段。

### 3.4 结果统计

结果统计主逻辑位于：

- `shared/questionModel.js`
- `backend/src/routes/surveys.js`
- `frontend/src/views/survey/SurveyAnswersPanel.vue`

后端当前已生成真实统计的题型包括：

- `radio`
- `checkbox`
- `input`
- `textarea`
- `slider`
- `rating`
- `scale`
- `matrix`
- `ranking`
- `upload`
- `date`
- `ratio`

对应统计能力如下：

| 题型 | 当前统计输出 |
|---|---|
| `radio / checkbox` | 选项人数、比例 |
| `input / textarea` | 样本答案 |
| `slider` | 平均值、最小值、最大值 |
| `rating / scale` | 平均分、分布 |
| `matrix` | 行维度下的选项分布 |
| `ranking` | 出现次数、比例、平均排名 |
| `upload` | 文件总数、文件样本 |
| `date` | 样本日期、最早/最晚日期 |
| `ratio` | 平均占比、非零比例、总占比 |

同时还会返回：

- `submissionTrend`
- `avgDuration`
- `today`
- `systemStats.devices`
- `systemStats.browsers`
- `systemStats.operatingSystems`
- `regionStats`

## 5. 仍然存在的题型相关缺口

虽然题型本身已经明显收口，但仍存在三类缺口。

### 5.1 注册表与页面实现仍未完全收口

表现：

- 共享注册表已经建立，但 `CreateSurveyPage.vue`、映射器、统计面板中仍保留不少分散判断
- 新题型虽然已打通，但“定义一次、多处自动消费”的目标还没彻底实现

影响：

- 后续继续扩展 `31 / 32` 等剩余题型时，仍可能出现同步遗漏

### 5.2 浏览器级回归仍缺位

表现：

- 接口级验证、系统 smoke、前端构建都已通过
- `npm run test:e2e` 仍返回 `Error: No tests found`

影响：

- 结果页图表、时间筛选、附件下载、SPSS 语法下载等真实交互仍缺浏览器自动化保护

### 5.3 分析页的数据链路仍有边界

表现：

- 趋势图、地域分布、系统环境图已经消费真实接口
- 但 `regionStats` 是否有正向数据，仍依赖答卷来源地采集是否真正落库

影响：

- 页面结构已不是占位，但地域图表在主数据库没有省市来源时仍只能展示空态

## 6. 是否适合继续进入“丰富题型阶段”

结论：

**不适合把下一阶段定义为“继续补齐矩阵多选 / 矩阵下拉 / 比重题”。**

更合理的命名应为：

**题型收口、分析体验增强与浏览器级回归阶段**

原因：

- `matrix multi / matrix dropdown / ratio` 已经不再是未交付项
- 当前主流题型闭环已经足够完整，瓶颈不再是“题型数量太少”
- 真正的短板在于统一建模、浏览器回归、地域正向数据和剩余评分族题型

## 7. 下一阶段建议

### 7.1 第一优先级：继续收口题型注册中心

建议继续把以下能力围绕共享注册表集中：

- 标签与图标
- legacy/server 映射
- 是否需要选项
- 默认配置
- 编辑器配置面板
- 填写页渲染
- 提交校验
- 结果统计适配

目标不是再造一层抽象，而是把当前已经分散在多个文件里的事实源收回到同一套配置。

### 7.2 第二优先级：补浏览器级回归

优先补齐：

- 登录
- 创建问卷
- 公开填写
- 结果页图表切换
- 时间筛选
- Excel / 附件 / SPSS 下载动作

### 7.3 第三优先级：补剩余评分族题型

推荐顺序：

1. `31` 评分单选
2. `32` 评分多选
3. 再评估是否需要更异构的扩展题型

原因：

- 这两类现在才是真正还未交付、且仍在现有题型族谱中的缺口
- 它们和现有 `rating / scale / option` 结构的复用潜力高

## 8. 总体判断

当前题型系统的真实状态可以概括为：

- 基础题型已稳定闭环
- `rating / scale / matrix / upload / ratio` 都已进入真实可用范围
- `matrix multi / matrix dropdown / ratio` 已从旧文档里的“未交付”推进为真实交付
- 统计接口与前端分析页已经形成共享统计模型和真实图表呈现
- 剩余重点从“能不能作答”转向“能不能长期维护、能不能被浏览器回归保护、能不能把真实数据看得更清楚”

因此，下一阶段最合理的目标不是：

**继续无差别丰富题型数量**

而是：

**把现有题型体系做成一套更统一、更可维护、更可验证、更可视化的交付能力**
