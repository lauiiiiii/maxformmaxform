# 问卷结构与题型能力复评
基于当前代码状态复评时间：2026-03-27

## 1. 复评结论

当前项目已经不适合再按“继续补基础题型”的阶段来描述。

更准确的判断是：

- 主流题型已经形成“创建 / 保存 / 填写 / 提交 / 统计”的闭环。
- 编辑器、填写页、结果页已经围绕同一套题型语义在协作。
- 当前短板主要不在“题型数量太少”，而在“问卷结构继续收口、剩余占位题型继续下线或补齐、自动化回归补齐、统计读模型补齐”。

一句话结论：

**问卷模块当前处于“结构收口与能力做实阶段”，不是“基础题型补全阶段”。**

## 2. 当前问卷结构的真实事实源

### 2.1 题型注册中心

题型的统一事实源是：

- `shared/questionTypeRegistry.js`

当前这里统一维护：

- legacy `uiType`
- server `type`
- 题型标签
- 是否实现
- 默认选项
- 配置面板类型
- 渲染类型
- 统计语义
- 提交语义

### 2.2 题目统计模型

题目级统计统一由：

- `shared/questionModel.js`

负责。

这意味着题型定义和题目统计，已经不是前后端各写一套了。

### 2.3 后端问卷结构规范

后端的问卷归一化与校验入口在：

- `backend/src/utils/questionSchema.js`

它负责：

- 题型归一化
- 选项归一化
- 矩阵行归一化
- 上传题配置归一化
- 发布前最小校验
- 提交答案校验

### 2.4 前端问卷结构封装

前端当前围绕共享注册表做了两层封装：

- `frontend/src/utils/questionTypeRegistry.ts`
- `frontend/src/utils/questionEditorModel.ts`

它们分别承担：

- 共享注册表的前端类型包装
- 编辑器创建默认题目的草稿模型

## 3. 当前问卷生命周期结构

### 3.1 创建与编辑

当前编辑器入口和拆分结构为：

- 引导页：`frontend/src/views/survey/CreateSurveyLanding.vue`
- 编辑入口：`frontend/src/views/survey/CreateSurveyPage.vue`
- 页面壳：`frontend/src/views/survey/CreateSurveyPageShell.vue`
- 题型面板：`frontend/src/views/survey/CreateSurveyQuestionConfigPanel.vue`
- 题目列表：`frontend/src/views/survey/CreateSurveyQuestionListPanel.vue`
- 逻辑弹层：`frontend/src/views/survey/CreateSurveyLogicSettings.vue`
- 发布与分享：`frontend/src/views/survey/CreateSurveyPublishPanel.vue`
- 业务编排：`frontend/src/composables/useSurveyEditor.ts`

这说明当前问卷结构已经不是“一个页面负责全部编辑行为”的模式。

### 3.2 填写

填写页结构为：

- PC 填写：`frontend/src/views/survey/FillSurveyPage.vue`
- 移动端填写：`frontend/src/views/survey/FillSurveyMobilePage.vue`

填写页关键能力已拆成独立 hook：

- 可见性：`useFillSurveyVisibility.ts`
- 跳题：`useFillSurveyJumpLogic.ts`
- 配额：`useFillSurveyQuota.ts`
- 上传：`useFillSurveyUpload.ts`

### 3.3 结果分析

结果页统计链路为：

- 后端：`backend/src/services/surveyService.js`
- 共享统计：`shared/questionModel.js`
- 前端展示：`frontend/src/views/survey/SurveyAnswersPanel.vue`、`QuestionAnalyticsChart.vue`

### 4.3 关于左侧题型面板的真实情况

编辑器左侧面板还保留了不少 legacy 入口，例如：

- 多项填空
- 分页符
- 部分矩阵衍生题
- AI 题型
- OCR / 录音 / 视频题

但当前 `useSurveyEditor.ts` 中的 `addQuestionByType()` 会在未实现题型上直接拦截，并提示“正在开发中”。

因此更准确的文档口径应是：

**面板展示了完整的 legacy 题型族谱，但真正进入主链路的仍只有共享注册表里标记为 `implemented=true` 的那一批。**

## 5. 当前问卷结构字段

当前单题对象并不是只有“标题 + 选项”。

问卷结构已经支持以下核心字段：

- 通用字段
  - `type`
  - `uiType`
  - `title`
  - `titleHtml`
  - `description`
  - `required`
  - `order`
  - `hideSystemNumber`
- 选择题相关
  - `options[]`
  - `optionOrder`
  - `optionGroups`
  - `groupOrderRandom`
  - `autoSelectOnAppear`
- 数值 / 评分题相关
  - `validation`
- 上传题相关
  - `upload.maxFiles`
  - `upload.maxSizeMb`
  - `upload.accept`
- 矩阵题相关
  - `matrix.selectionType`
  - `matrix.rows[]`
- 逻辑相关
  - `logic.visibleWhen`
  - `jumpLogic`
  - `options[].visibleWhen`
- 配额相关
  - `quotasEnabled`
  - `quotaMode`
  - `quotaFullText`
  - `quotaShowRemaining`

这已经不是“简单问卷结构”，而是具备编辑器编排语义的问卷结构。

## 6. 当前数据流判断

### 6.1 保存链路

编辑器内部仍以 legacy 题型为主，但在保存时会转换为：

- server `type`
- legacy `uiType`
- 结构化 `options / matrix / validation / upload / logic / jumpLogic`

这条转换链主要位于：

- `frontend/src/composables/useSurveyEditor.ts`

### 6.2 提交链路

答卷提交由以下链路负责：

- 前端组装：`FillSurveyPage.vue`
- 后端校验：`validateSubmissionAnswers()`
- 上传解析：`uploadSurveyFile()` + `resolveUploadAnswers()`
- 最终提交：`submitSurveyResponse()`

### 6.3 统计链路

结果统计当前仍是“按答卷实时聚合”：

- 数据源：`answers.answers_data`
- 题目统计：`shared/questionModel.js`
- 问卷汇总：`surveyService.getSurveyResults()`

## 7. 当前阶段最重要的问题

### P1. 题型结构已经收口，但编辑器编排还未继续拆开

题型定义已经开始统一，但编辑器侧的大量逻辑仍集中在：

- `frontend/src/composables/useSurveyEditor.ts`

因此下一步最应该继续治理的是“编辑器编排层”，而不是再泛化新增更多题型入口。

### P1. 题型面板与真实能力边界仍存在展示偏差

虽然未实现题型会被拦截，但左侧面板仍向用户暴露了大量未落地入口。

这带来的问题是：

- 用户以为某题型已支持
- 文档很容易跟着 UI 文案误判
- 后续回归时容易把“面板可见”误当成“题型闭环”

### P1. 统计仍依赖实时扫描 JSON 答卷

当前题型闭环已经够用，但统计层仍然是：

- 实时聚合
- 无读模型
- 无统计快照

在题量或答卷量继续上升后，这会是问卷结构层面最真实的瓶颈。

### P2. 地域统计仍然是边界能力，不是稳定主结构

`regionStats` 当前仍依赖答卷来源字段是否存在。

所以这块更适合在文档里表述为：

- 已预留统计出口
- 但字段采集链路还未完全稳定

而不是把它描述成完全成熟的主能力。

## 8. 更新后的阶段命名

旧命名“丰富题型阶段”已经不够准确。

更合适的阶段命名是：

**问卷结构收口、剩余题型补齐与回归治理阶段**

原因是：

- 已实现题型已经覆盖主链路
- 当前真正的短板是结构治理与验证，而不是单纯增加题型数量

## 9. 下一步建议

### 第一优先级

- 收敛左侧题型面板，只展示已实现题型，未实现题型移入“规划中”
- 拆分 `useSurveyEditor.ts`
- 统一问卷结构字段说明和保存映射文档

### 第二优先级

- 为结果页补统计快照或读模型
- 为编辑器与填写页补浏览器级回归

### 第三优先级

- 补齐 `31 / 32`
- 再决定是否继续打开更异构的题型族

## 10. 文档口径修正

本次复评后，问卷相关文档应统一采用以下判断：

- 题型系统已进入主链路闭环阶段
- 编辑器与填写页已完成第一轮结构拆分
- 剩余重点是收口结构、补自动化和继续做实共享模型
