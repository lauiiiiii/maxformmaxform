## 4. 额外观察

### 4.1 并行执行两个 smoke 入口会失败

今天额外做了一次并行实验：

- `node scripts/system-smoke.mjs`
- `cmd /c npm run smoke:system`

并行执行结果：

- 后端启动报错：`EADDRINUSE: address already in use :::63102`
- 前端 npm smoke 返回：`TypeError: fetch failed`

结论：

- 两个 smoke 入口共用固定端口 `63102` 和共享测试库
- 当前只能顺序执行，不能并行验收

### 4.2 Playwright WebServer 仍有警告

浏览器回归虽然通过，但启动日志里仍出现：

- `util._extend` 废弃警告
- `defineProps` / `defineEmits` 编译器宏导入警告

这些不影响本轮通过结论，但值得后续清理。

## 6. 风险与缺口

- 流程与题库仓库前端页面仍依赖占位 API，尚未进入真实接口验证
- smoke 仍依赖固定端口与共享测试库，不适合并行或隔离 CI
- Playwright 当前虽已扩到 `8` 条，但还未覆盖答卷下载、消息中心、文件夹管理和更多后台操作
- 高级分析 / SPSS 仍未形成真实系统级回归

## 7. 建议

1. 若要支持并行 CI，优先改造 smoke 的端口与测试库隔离。
2. 为流程与题库仓库补真实后端接口，再补一轮 API 和 E2E 文档。
3. 将 Playwright 继续扩到下载、消息、文件夹与更多权限边界场景。
4. 清理前端 WebServer 启动警告，降低回归日志噪音。
