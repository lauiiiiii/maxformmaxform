# 开发日志 - 2026-03-28

## 目标

今天这轮工作不是继续补单一题型，而是基于 `2026-03-26` 的源码状态，把系统从“题型与结果页增强”推进到“服务化分层、编辑器解耦、浏览器回归落地、文档口径对齐”的下一阶段。

本轮目标有四项：

1. 把问卷、答卷和后台管理域从大量 route 内聚逻辑，推进到更明确的 `service / repository / contract` 分层。
2. 把问卷编辑器从巨型单页推进到“页面壳 + 子面板 + 领域 composable”的真实结构。
3. 把 `2026-03-26` 仍处于缺位状态的 Playwright 浏览器级回归落地，并补强系统冒烟。
4. 基于最新源码与实测结果，刷新 README、开发指南、测试报告和结构评估文档。

## 今日结论

- 相比 `2026-03-26`，本轮最大的变化不再是单点题型交付，而是系统结构进入服务化和契约化阶段。
- `Playwright` 已从“有命令但无用例”变为“有真实浏览器回归”。
- 问卷编辑器已经完成一轮明确的结构拆分，不再适合继续按单页巨石理解。
- 后端结果统计已经具备快照与最小可观测字段，不再只是单纯实时聚合。
- 当前文档口径也已从旧的“巨型路由 / Playwright 未落地 / smoke 56 项”刷新到当前真实状态。

## 当前遗留问题

- 管理域虽然已经开始服务化，但不同模块的访问控制、事务边界和聚合模式还不完全统一。
- `editorContext` 暴露字段仍然偏多，页面壳与子面板之间的依赖面还可以继续收缩。
- 系统冒烟使用固定端口与共享测试库，不适合并行跑多个实例。
- 地域统计的正向数据链路仍不稳定，常见结果仍然是空态。
- 浏览器 E2E 目前只覆盖 3 条主链路，后台管理、上传题、权限边界还没有纳入。

## 下一步建议

1. 继续把用户、角色、部门、文件夹等管理域向更完整的 `policy + service + repository` 收口。
2. 把 Playwright 用例扩展到管理后台、上传题、回收站和权限场景。
3. 继续缩小 `editorContext` 对外暴露面，减少页面壳和子面板的耦合。
4. 为结果快照补更稳定的命中率观察口径和更明确的大样本基线指标。
