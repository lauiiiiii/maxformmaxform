# 问卷结构与题型规范
基于当前代码状态更新时间：2026-03-27

## 1. 文档目标

本文档只描述当前仓库里已经存在的真实实现，用于统一以下问题：

- 问卷如何存储
- 题型如何映射
- 编辑器如何构造题目
- 提交时后端如何校验
- 结果页如何统计

若其他历史文档与当前代码冲突，以本文件和当前代码为准。

## 2. 当前结构原则

### 2.1 单一事实源

当前问卷领域的单一事实源是：

- 题型定义：`shared/questionTypeRegistry.js`
- 题目统计：`shared/questionModel.js`
- 后端结构归一化与校验：`backend/src/utils/questionSchema.js`
- 前端编辑器草稿模型：`frontend/src/utils/questionEditorModel.ts`

### 2.2 持久化原则

当前项目采用：

- `surveys.questions` 保存问卷题目 JSON
- `answers.answers_data` 保存答卷 JSON

这是一套“写模型灵活、统计侧后补”的结构，而不是严格拆成独立 `Question` 表。

## 3. 服务端题型集合

当前后端正式识别的服务端题型为：

- `input`
- `textarea`
- `radio`
- `checkbox`
- `date`
- `slider`
- `ranking`
- `upload`
- `rating`
- `scale`
- `matrix`
- `ratio`

这些类型的归一化和校验由 `questionSchema.js` 负责。

## 5. 单题对象结构

当前 `surveys.questions[]` 中的单题对象，采用宽松 JSON 结构。

一个典型示例如下：

```json
{
  "id": "q_xxx",
  "type": "matrix",
  "uiType": 21,
  "title": "请评价以下维度",
  "titleHtml": "<p>请评价以下维度</p>",
  "description": "可补充填写说明",
  "required": true,
  "order": 1,
  "hideSystemNumber": false,
  "options": [
    { "label": "非常满意", "value": "1", "order": 1 },
    { "label": "满意", "value": "2", "order": 2 }
  ],
  "matrix": {
    "selectionType": "multiple",
    "rows": [
      { "label": "服务态度", "value": "1", "order": 1 },
      { "label": "响应速度", "value": "2", "order": 2 }
    ]
  },
  "validation": {},
  "logic": {
    "visibleWhen": []
  },
  "jumpLogic": {
    "byOption": { "1": "5" }
  }
}
```

## 6. 当前字段规范

### 6.1 通用字段

- `type`
  - 服务端题型
- `uiType`
  - legacy 题型编号
- `title`
  - 纯文本标题
- `titleHtml`
  - 富文本标题
- `description`
  - 题目说明
- `required`
  - 是否必答
- `order`
  - 题目顺序
- `hideSystemNumber`
  - 是否隐藏系统题号

### 6.2 选项题字段

- `options[]`
  - `label`
  - `value`
  - `order`
  - `rich`
  - `desc`
  - `exclusive`
  - `defaultSelected`
  - `hidden`
  - `fillEnabled`
  - `fillRequired`
  - `fillPlaceholder`

  - `quotaEnabled`
  - `visibleWhen`

- `validation.min`

### 6.6 逻辑与配额字段

## 7. 编辑器保存规范

当前前端编辑器内部仍大量使用 legacy 题型表达，但在保存到后端时会做两件事：

1. 将编辑器题型转换为服务端 `type`
2. 同时保留 `uiType` 作为前端还原语义

也就是说：

- `type` 用于后端持久化和提交校验
- `uiType` 用于前端恢复 legacy 编辑语义和特殊渲染

## 9. 提交规范

### 9.1 提交载荷

当前答卷提交结构为：

```json
{
  "answers": [
    {
      "questionId": 1,
      "value": "1"
    }
  ],
  "duration": 35,
  "clientSubmissionToken": "token-xxx"
}
```

说明：

- `questionId` 当前采用题目顺序号
- 后端提交后会补齐 `questionType`

### 9.2 标准化后的答卷结构

标准化后写入 `answers.answers_data[]` 的结构为：

```json
[
  {
    "questionId": 1,
    "questionType": "radio",
    "value": "1"
  }
]
```

### 9.3 当前已覆盖的提交校验

后端当前已覆盖：

- 单选值合法性
- 多选 / 排序数组结构
- 数值题数值合法性
- 评分题上下界校验
- 矩阵行列结构校验
- 比重题 `0-100` 与总和 `100` 校验
- 上传题文件数量 / 大小 / 类型校验
- 必填校验

## 10. 上传题规范

当前上传题采用两阶段提交。

### 10.1 上传阶段

接口：

- `POST /api/surveys/:id/uploads`

后端会：

- 校验目标题目是否为上传题
- 校验文件大小与类型
- 记录待提交文件
- 返回 `uploadToken`

### 10.2 正式提交阶段

正式提交时，客户端只提交文件引用：

```json
{
  "questionId": 10,
  "value": [
    {
      "id": 101,
      "uploadToken": "token-xxx"
    }
  ]
}
```

后端再根据：

- `submissionToken`
- `id`
- `uploadToken`

解析真实文件，并补齐：

- `name`
- `url`
- `size`
- `type`

## 11. 统计规范

当前结果页统计不依赖外部分析库，而是直接聚合答卷 JSON。

### 11.1 题目级统计

| 题型 | 统计输出 |
|---|---|
| `radio` / `checkbox` | 选项人数、占比 |
| `ranking` | 出现次数、占比、平均排名 |
| `ratio` | 选择人数、占比、平均占比、总占比 |
| `input` / `textarea` | 样本答案 |
| `date` | 样本日期、最早、最晚 |
| `slider` | 平均值、最小值、最大值 |
| `rating` / `scale` | 平均分、分布 |
| `matrix` | 行维度下的选项分布 |
| `upload` | 文件总数、样本文件 |

### 11.2 问卷级汇总

结果接口还会返回：

- `totalSubmissions`
- `completed`
- `incomplete`
- `completionRate`
- `submissionTrend`
- `avgDuration`
- `today`
- `systemStats.devices`
- `systemStats.browsers`
- `systemStats.operatingSystems`
- `regionStats`

说明：

- `regionStats` 当前仍依赖提交来源字段是否存在，属于边界能力，不应视为完全成熟主结构

## 12. 新增题型的最低接入要求

一个题型要被认定为“正式支持”，至少要同步补齐以下位置：

1. `shared/questionTypeRegistry.js`
2. `shared/questionModel.js`
3. `backend/src/utils/questionSchema.js`
4. `frontend/src/utils/questionTypeRegistry.ts`
5. `frontend/src/utils/questionEditorModel.ts`
6. 编辑器面板与题目列表
7. 填写页
8. 结果页统计展示

最低标准不是“面板里能点出来”，而是：

- 能创建
- 能保存
- 能恢复
- 能填写
- 能提交
- 能统计，或明确声明不统计

## 13. 当前阶段判断

当前问卷结构最合理的判断是：

- 基础题型体系已成型
- 共享题型模型已建立
- 编辑器与填写页已完成第一轮结构拆分
- 后续重点应放在结构继续收口、统计读模型、自动化回归和剩余题型补齐
