# 问卷题型规范

基于源码状态更新时间：2026-03-26

本文档以当前仓库代码为准，用于描述题型系统的真实实现方式。若其它文档仍提到 MongoDB、`Question._id`、ClickHouse 聚合等历史方案，以本文和当前源码为准。

## 一、规范目标

本文关注的是当前真实实现，而不是理想蓝图，核心回答以下问题：

- 题目如何在后端保存
- legacy 题型如何映射到服务端题型
- 哪些题型已经进入真实可用范围
- 提交答卷时后端如何标准化与校验
- 结果页当前能输出哪些统计结构

## 二、当前架构事实

### 2.1 数据存储

当前项目采用 `MySQL + JSON 字段` 方案：

- 问卷定义存放在 `surveys` 表
- 题目列表整体存放于 `surveys.questions`
- 答卷存放在 `answers` 表
- 答案明细整体存放于 `answers.answers_data`

与题型直接相关的上传附件记录放在 `files` 表，并通过以下字段与问卷/答卷绑定：

- `survey_id`
- `question_order`
- `submission_token`
- `public_token`
- `answer_id`

当前不存在以下历史结构：

- 独立的 `Question` 表
- `Question._id` 作为提交流程主键
- ClickHouse 作为正式结果统计链路

### 2.2 单一事实源

题型定义的权威来源在：

- `shared/questionTypeRegistry.js`

该文件统一维护：

- 服务端题型 `type`
- legacy 题型 `uiType`
- `analyticsKind`
- `submissionKind`
- 是否需要选项
- 默认选项 / 默认配置
- 特殊渲染类型

结果统计的共享模型在：

- `shared/questionModel.js`

## 三、服务端题型

当前后端识别的服务端题型共有 12 类：

- `input`
- `textarea`
- `radio`
- `checkbox`
- `date`
- `slider`
- `ranking`
- `upload`
- `rating`
- `scale`
- `matrix`
- `ratio`

这些题型的标准化与提交流程校验由 `backend/src/utils/questionSchema.js` 负责。

## 四、legacy 题型映射

### 4.1 已实现的 legacy 题型

当前注册表里标记为 `implemented` 的 legacy 题型为：

| legacy | 名称 | 服务端题型 | 说明 |
|---|---|---|---|
| 1 | 填空题 | `input` | 默认题型 |
| 2 | 简答题 | `textarea` | 文本题 |
| 3 | 单选题 | `radio` | 选项题 |
| 4 | 多选题 | `checkbox` | 选项题 |
| 7 | 下拉题 | `radio` | 通过 `uiType=7` 区分渲染 |
| 8 | 滑动条题 | `slider` | 数值题 |
| 11 | 排序题 | `ranking` | 数组答案 |
| 13 | 文件上传 | `upload` | 两阶段提交 |
| 14 | 日期 | `date` | 日期字符串 |
| 18 | 段落说明 | `input` | 通过 `renderType=stage_explain` 跳过统计 |
| 20 | 矩阵单选 | `matrix` | `selectionType=single` |
| 21 | 矩阵多选 | `matrix` | `selectionType=multiple` |
| 24 | 矩阵下拉 | `matrix` | `renderType=matrix_dropdown` |
| 29 | 星级题 | `rating` | 有上下界 |
| 30 | NPS 量表 | `scale` | 有上下界与标签 |
| 34 | 评价题 | `scale` | `scale` 的历史别名 |
| 36 | 比重题 | `ratio` | 百分比分配题 |

当前共享注册表中仍未正式实现的主要 legacy 题型包括：

- `31` 评分单选
- `32` 评分多选

### 4.2 关键映射规则

- 下拉题不是独立服务端类型，而是 `radio + uiType=7`
- 段落说明不是独立服务端类型，而是通过 `uiType=18` / `renderType=stage_explain` 表示
- 矩阵下拉不是独立服务端类型，而是 `matrix + uiType=24 + renderType=matrix_dropdown`
- 矩阵多选不是独立服务端类型，而是 `matrix + selectionType=multiple`
- 未知题型会回退为 `input`

## 五、问题对象结构

`surveys.questions[]` 中的单题对象当前采用宽松 JSON 结构，常见字段如下：

```jsonc
{
  "id": "frontend-generated-id",
  "type": "matrix",
  "uiType": 21,
  "title": "题目标题",
  "titleHtml": "<p>富文本标题</p>",
  "description": "题目说明",
  "required": false,
  "order": 1,
  "options": [
    {
      "label": "选项1",
      "value": "1",
      "order": 1
    }
  ],
  "validation": {},
  "upload": {
    "maxFiles": 1,
    "maxSizeMb": 10,
    "accept": ".jpg,.jpeg,.png,.gif,.webp,.pdf,.docx,.xlsx"
  },
  "matrix": {
    "selectionType": "multiple",
    "rows": [
      { "label": "维度1", "value": "1", "order": 1 }
    ]
  }
}
```

补充说明：

- `type` 是后端持久化使用的服务端题型
- `uiType` 是前端编辑器语义补充字段
- `options`、`upload`、`matrix` 会在后端保存前标准化
- `ratio` 的答案结构为对象，键是选项值，值是百分比数字
- 后端并不依赖严格 schema，但会在保存、发布、提交阶段做最小校验

## 六、保存与读取规范

### 6.1 前端保存到后端

前端编辑器保存问卷时：

- 将编辑器内部的 legacy 类型转换为服务端 `type`
- 同时保留 `uiType`
- 题目顺序最终按 `order` 落库

例如：

```jsonc
{
  "type": "matrix",
  "uiType": 24,
  "matrix": {
    "selectionType": "single"
  }
}
```

### 6.2 后端标准化保存

创建和更新问卷时，后端会通过 `normalizeSurveyQuestions()` 做标准化：

- 统一标准化题型
- 统一补齐 `order`
- 统一标准化 `options`
- 统一标准化 `upload`
- 统一标准化 `matrix.rows`
- 统一收敛矩阵选择模式为 `single` 或 `multiple`

### 6.3 前端读取还原

前端读取问卷后，会通过 `mapServerTypeToLegacy(type, uiType)` 把服务端题型还原回编辑器可识别的 legacy 类型。

优先级：

1. 优先使用 `uiType`
2. 若缺失，再根据服务端 `type` 回推

## 八、答卷提交规范

### 8.1 基础提交结构

当前前端提交答卷的结构为：

```jsonc
{
  "answers": [
    {
      "questionId": 1,
      "value": "A"
    }
  ],
  "duration": 35,
  "clientSubmissionToken": "..."
}
```

说明：

- `questionId` 当前是题目顺序号，1-based
- 后端不会映射到独立题目表主键
- 标准化后会补充 `questionType`

### 8.2 标准化后的答案结构

后端通过 `validateSubmissionAnswers()` 将答案标准化为：

```jsonc
{
  "questionId": 1,
  "questionType": "radio",
  "value": "1"
}
```

因此 `answers.answers_data[]` 的事实结构为：

```jsonc
[
  {
    "questionId": 1,
    "questionType": "radio",
    "value": "1"
  },
  {
    "questionId": 2,
    "questionType": "ratio",
    "value": { "brand": 40, "price": 35, "service": 25 }
  }
]
```

### 8.3 当前已覆盖的提交校验

后端当前已实现的提交校验包括：

- `radio` 单值选项合法性校验
- `checkbox / ranking` 数组结构与选项合法性校验
- `slider` 数值校验
- `rating / scale` 数值范围校验
- `matrix(single)` 行键与单值选项校验
- `matrix(multiple)` 行键与数组选项校验
- `ratio` 对象结构、数字合法性、`0-100` 范围与合计 `100` 校验
- `upload` 数组结构、文件数量、大小、类型校验
- `required` 必填校验

## 九、上传题规范

上传题采用两阶段提交。

### 9.1 上传阶段

接口：

- `POST /api/surveys/:id/uploads`

请求中通常包含：

- 文件本体
- `questionId`
- `submissionToken`

后端会：

- 校验问卷已发布
- 校验目标题目确实是上传题
- 校验数量、大小、扩展名/MIME
- 将文件记录为待提交状态
- 返回 `uploadToken`

### 9.2 提交答卷阶段

正式提交上传题答案时，客户端只需要传最小文件引用：

```jsonc
{
  "questionId": 10,
  "value": [
    {
      "id": 101,
      "uploadToken": "token-xxx"
    }
  ]
}
```

后端会根据 `submissionToken + id + uploadToken` 解析真实文件记录，再校验：

- 文件是否属于当前问卷
- 文件是否属于当前题号
- 文件是否属于当前提交会话
- 数量 / 大小 / 类型是否满足上传题配置

最终落库的上传题答案会补齐：

- `name`
- `url`
- `size`
- `type`

## 十、发布校验规范

当前发布问卷时，后端会通过 `validateSurveyQuestions()` 做最小校验：

- 问卷至少一题
- 每题必须有标题
- 所有带选项题型至少两个选项
- `matrix` 至少一个行、至少两个列

需要注意：

- 这仍是“当前实现可保证的最小校验”
- 并不是完整 schema 驱动的题型发布校验体系

## 十一、结果统计规范

当前结果接口：

- `GET /api/surveys/:id/results`

统计逻辑不走 ClickHouse，而是直接基于 `answers.answers_data` 聚合，并复用 `shared/questionModel.js`。

### 11.1 当前支持的统计输出

| 题型 | 统计输出 |
|---|---|
| `radio` / `checkbox` | `options[].count / percentage` |
| `ranking` | `options[].count / percentage / avgRank` |
| `ratio` | `options[].count / percentage / totalShare / avgShare` |
| `input` / `textarea` | `sampleAnswers / totalAnswers` |
| `date` | `sampleAnswers / earliestDate / latestDate` |
| `slider` | `avgValue / minValue / maxValue` |
| `rating` / `scale` | `avgScore / distribution` |
| `matrix` | `rows[].options[].count / percentage` |
| `upload` | `totalFiles / sampleFiles / totalAnswers` |

### 11.2 题目之外的汇总统计

结果接口还会返回：

- `totalSubmissions`
- `completed`
- `incomplete`
- `completionRate`
- `submissionTrend`
- `avgDuration`
- `today`
- `systemStats.devices`
- `systemStats.browsers`
- `systemStats.operatingSystems`
- `regionStats`

其中 `regionStats` 的正向数据依赖答卷来源地字段，当前主链路还未稳定落地省市采集，因此常见为空态结构。

### 11.3 统计跳过规则

- `stage_explain` 说明块不会进入题目统计

## 十二、当前题型状态划分

### 12.1 已形成主链路闭环

以下题型当前已经具备“编辑 / 保存 / 填写 / 提交 / 统计”的真实闭环：

- `input`
- `textarea`
- `radio`
- `checkbox`
- `dropdown`
- `slider`
- `ranking`
- `upload`
- `date`
- `rating`
- `scale`
- `matrix(single / multiple / dropdown)`
- `ratio`

补充说明：

- `stage_explain` 可视为编辑/渲染闭环，但不参与答卷统计

### 12.2 未正式支持或仍属占位

以下题型不应视为正式支持：

- `31` 评分单选
- `32` 评分多选
- 录音 / OCR / 视频等扩展题
- AI 题型占位能力

## 十三、新增题型的正确流程

后续如需新增题型，建议至少同步处理以下文件：

1. `shared/questionTypeRegistry.js`
2. `shared/questionModel.js`
3. `backend/src/utils/questionSchema.js`
4. `frontend/src/utils/questionTypeRegistry.ts`
5. `frontend/src/views/survey/CreateSurveyPage.vue`
6. `frontend/src/views/survey/FillSurveyPage.vue`
7. `frontend/src/views/survey/FillSurveyMobilePage.vue`
8. `frontend/src/views/survey/SurveyAnswersPanel.vue`

新增题型达到“正式支持”的最低标准应包括：

- 编辑器可创建
- 保存后可稳定还原
- 填写页有明确交互
- 提交时有题型校验
- 结果页有统计适配，或明确声明不做统计
