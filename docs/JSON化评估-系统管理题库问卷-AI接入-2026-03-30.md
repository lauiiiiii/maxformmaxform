# AI 接入现状重评估：系统管理、题库、问卷参数与问卷逻辑

更新时间：2026-03-31

## 1. 评估范围

本次重新评估基于当前仓库真实代码，重点回答四个问题：

- 系统管理是否已经具备受控的 AI 执行动作协议
- 题库是否已经具备可供 AI 使用的结构化内容承载能力
- 问卷参数是否已经形成稳定的 JSON 输入输出边界
- 问卷逻辑是否已经具备共享类型、后端校验和 dry-run 能力

本次结论以当前代码为准，不沿用旧版本文档中的历史假设。

## 2. 总体结论

当前系统的 AI 接入状态，不应再概括为“准备阶段”。

更准确的判断是：

- 系统管理 AI：已经落地第一版正式接入
- 问卷 JSON AI：已经具备受控导入和 dry-run 校验能力
- 问卷逻辑 AI：已经具备共享类型、清洗和后端结构校验
- 题库 AI：已经有结构化内容容器，但还没有收口成统一强约束 DSL

因此，当前系统最适合推进的不是“重新讨论要不要接 AI”，而是分层推进：

1. 继续做实管理域 AI 动作协议
2. 将问卷 JSON 导入升级为真正的服务端 AI 生成链路
3. 把题库内容容器继续收口为统一题目 DSL

## 3. 重新评估后的分项结论

### 3.1 系统管理 AI：已正式落地，不再只是建议方案

这一块与旧文档相比变化最大。

当前已经存在完整的管理域 AI 协议与执行链路：

- 后端路由：`/api/management-ai/protocol`
- 执行入口：`/api/management-ai/actions`
- 执行账本：`/api/management-ai/executions`
- 账本导出：`/api/management-ai/executions/export`
- 前端入口：`frontend/src/views/admin/Config.vue`

当前管理域 AI 具备以下能力：

- 统一协议版本 `2026-03-31`
- 统一动作 envelope：`management.action`
- `dryRun` 校验
- `idempotencyKey` 幂等执行
- 执行结果回放
- 失败请求回填
- 执行账本查询与导出
- 管理员权限边界
- 通过现有 service 执行，而不是直接写表
- 审计与消息记录

当前已支持的动作类型包括：

- 用户：创建、更新、删除、重置密码
- 角色：创建、更新、删除
- 部门：创建、更新、删除
- 岗位：创建、更新、删除
- 流程：创建、更新、删除
- 题库仓库：创建、更新、删除
- 题库题目：创建、删除

结论：

**系统管理 AI 已经从“可设计”进入“已接入、可执行、带审计边界”的阶段。**

### 3.2 问卷参数 JSON：已经成熟，可作为 AI 的标准输入输出

这部分依然是当前最成熟的 AI 接入基础。

当前问卷参数具备以下事实：

- 前后端共享题型映射与题目 DTO
- 编辑器保存时输出的是标准问卷 JSON
- 后端创建、更新、发布前都会走统一标准化和校验
- 前端存在 `validate` 与 `dry-run` 接口

现有能力包括：

- 问卷草稿校验：`POST /api/surveys/validate`
- 问卷 JSON dry-run：`POST /api/surveys/dry-run`
- 问卷字段清洗：共享 `survey.contract`
- 题型标准化与提交校验：`backend/src/utils/questionSchema.js`

当前已经明确结构化的问卷字段包括：

- `type / uiType`
- `title / titleHtml / description`
- `required / order / hideSystemNumber`
- `options`
- `matrix`
- `validation`
- `upload`
- `logic.visibleWhen`
- `jumpLogic`
- `optionGroups`
- `quotasEnabled / quotaMode / quotaShowRemaining / quotaFullText`
- `autoSelectOnAppear`
- `examConfig`

结论：

**问卷参数层已经具备作为 AI 标准 JSON 协议的条件，适合继续加强而不是推倒重来。**

### 3.3 问卷逻辑：不再是“类型松散、后端不校验”的状态

旧文档在这一点上已经过期。

当前问卷逻辑已经具备三层保护：

- 共享类型层：`shared/survey.contract.d.ts`
- 写入清洗层：`shared/survey.contract.js`
- 后端结构校验层：`backend/src/utils/questionSchema.js`

当前已经被显式建模或校验的逻辑结构包括：

- `logic.visibleWhen`
- 选项级 `visibleWhen`
- `jumpLogic.byOption`
- `jumpLogic.unconditional`
- `optionGroups`
- `quotaMode`
- `autoSelectOnAppear`

后端当前已校验的关键约束包括：

- `visibleWhen` 必须引用前置题目
- 操作符必须在受支持集合内
- 依赖选项值必须真实存在
- 跳题目标必须跳向后续题目或 `end`
- `optionGroups` 必须合法且不能重叠
- 配额和自动选择等能力只能用于支持的题型

结论：

**问卷逻辑已经具备“可供 AI 安全生成后再由服务端校验”的基础条件。**

仍然需要注意的是：

- 逻辑目前主要校验结构合法性，不是语义最优性
- 复杂逻辑组合仍需 dry-run 和人工复核

### 3.4 题库：已经有结构化内容容器，但还没有形成统一强 DSL

这一部分也需要修正文档结论。

旧文档认为题库只有：

- `title`
- `type`
- `difficulty`
- `score`

这个判断已经不准确。

当前题库除了基础列，还支持通过 `content` 承载结构化内容，并在 DTO 中对外展开为：

- `stem`
- `options`
- `correctAnswer`
- `analysis`
- `tags`
- `knowledgePoints`
- `applicableScenes`
- `aiMeta`
- `surveyQuestion`

其中：

- `surveyQuestion` 可以承载与问卷题接近的结构，用于题库题导入编辑器
- `aiMeta` 可以记录 AI 生成来源与审核状态
- 前端已支持从问卷导出当前题目回题库，并附带标签、知识点、场景和 AI 元数据

这说明题库已经不是“纯轻量索引目录”，而是进入了“结构化内容容器”的阶段。

但它仍然没有达到问卷 DSL 的成熟度，主要问题在于：

- `content` 允许承载较多自由字段
- `surveyQuestion` 当前只要求是对象，还不是强类型协议
- 缺少与问卷题相同级别的标准化和完整后端约束
- 缺少题库题的版本、审核、可发布状态等治理字段

结论：

**题库已经具备 AI 可用的结构化基础，但仍处于“半结构化内容容器”阶段，不宜宣称为成熟题目 DSL。**

## 4. 当前实际可用的 AI 接入方式

### 4.1 管理域 AI 接入

当前已经可以：

- 获取协议
- 生成受控动作 JSON
- 先 `dryRun`
- 再携带 `idempotencyKey` 执行
- 查询执行账本
- 导出账本
- 回填失败请求重新执行

适合当前落地的场景：

- 批量管理动作的 AI 编排
- 管理后台的 AI 助手
- 审批前的 AI 方案生成与 dry-run 校验

### 4.2 问卷 AI 接入

当前存在两条链路：

- 编辑器中的 AI JSON 导入
- 服务端 `validate` / `dry-run`

需要明确边界：

- 编辑器中的“AI 助手”本质上是前端 JSON 解析导入，不是后端 LLM 接口
- 当前系统已经允许把 AI 产出的问卷 JSON 送入服务端做 dry-run 校验
- 因此，真正缺的不是问卷 JSON 协议，而是“模型调用层”和“生成任务编排层”

### 4.3 题库 AI 接入

当前已经可以：

- 让 AI 生成题干、选项、标签、知识点、场景等结构化字段
- 将问卷题导出为题库题
- 从题库题回灌到问卷编辑器

当前还不适合：

- 直接让 AI 大规模自动沉淀题库且不经审核
- 把题库当成完全统一、强约束、可直接复用到任意题型的标准 DSL

## 5. 当前不应继续使用的旧判断

以下说法已不再准确：

- “系统管理还只是适合做 JSON 指令层，尚未真正接入 AI”
- “问卷逻辑类型约束和后端校验不够强”
- “题库题目目前只有 title/type/difficulty/score 四个轻量字段”

更准确的替代说法应为：

- 系统管理 AI 指令层已经落地并具备执行闭环
- 问卷逻辑已经有共享 contract、清洗和服务端结构校验
- 题库已具备结构化内容容器，但尚未升级为统一强 DSL

## 6. 下一步建议

### 6.1 第一优先级：把问卷 AI 从“前端导入”升级为“服务端正式链路”

建议新增：

- 服务端问卷生成接口
- Prompt 模板与输出协议版本
- 与 `POST /api/surveys/dry-run` 联动的自动校验
- 生成结果审计字段

目标：

AI 直接产出标准问卷 JSON，服务端完成校验、审计、落库前确认。

### 6.2 第二优先级：继续增强管理域 AI

截至 2026-03-31，管理域 AI 已经具备这些基础能力：

- 单动作协议 `management.action`
- `dryRun -> execute` 的受控执行路径
- `idempotencyKey` 幂等保护
- 单动作、单服务事务回滚边界
- 执行账本查询、导出、失败回填

下一阶段不应推翻这套协议，而应在此之上继续扩展。

#### 6.2.1 批量动作协议

当前协议明确要求“每次请求只执行一个管理动作”。这对第一阶段是对的，因为它把验证、审计、幂等、回放都收敛到了最小闭环。

但如果目标是企业可用的 AI 管理助手，就需要让 AI 一次提交一个“动作包”，而不是人工把十几个单动作逐个粘贴执行。

建议新增一个批量协议层，例如：

```json
{
  "kind": "management.batch",
  "version": "2026-04-xx",
  "batchId": "ops-20260401-001",
  "dryRun": true,
  "idempotencyKey": "batch-ops-20260401-001",
  "mode": "serial",
  "continueOnError": false,
  "actions": [
    {
      "stepId": "step-1",
      "action": {
        "kind": "management.action",
        "version": "2026-03-31",
        "action": "question_bank.repo.create",
        "input": {
          "name": "门店巡检题库"
        }
      }
    },
    {
      "stepId": "step-2",
      "action": {
        "kind": "management.action",
        "version": "2026-03-31",
        "action": "question_bank.question.create",
        "target": {
          "repoIdFrom": "step-1.result.id"
        },
        "input": {
          "title": "门店环境是否整洁？",
          "type": "radio"
        }
      }
    }
  ]
}
```

设计原则：

- 批量协议只是“动作容器”，单步动作仍然复用现有 `management.action`
- 第一版只支持 `serial` 串行模式，不急于开放并行
- 批量也必须有总幂等键，每个子步骤还应派生子幂等键
- 显式声明 `continueOnError`，不要让失败后的继续策略靠模型猜
- 步骤之间允许引用前一步输出，但只能引用协议暴露的稳定字段

这样做的好处是：

- 不破坏现有单动作执行器
- 批量能力可以由 orchestrator 层新增，而不是改写每个 service
- AI 可以生成“管理计划”，后台再决定是 dry-run、审批还是执行

#### 6.2.2 多步骤事务编排

批量动作并不等于“大事务”。这一点必须在设计上写死。

当前系统的事务边界是合理的：**一次动作，进入一个既有 service，在该 service 内完成主数据、审计、消息等一致性写入。**

下一阶段建议把事务分成两层：

- 本地事务层：继续保留现有“单动作、单服务事务”
- 编排层：为多步骤动作引入 saga / compensation 风格的状态机，而不是跨服务数据库大事务

建议的编排状态至少包括：

- `pending`
- `running`
- `completed`
- `failed`
- `compensating`
- `compensated`
- `partially_completed`

建议的编排规则：

- 如果多个步骤都落在同一个 service 且天然属于一个聚合写入，可继续在该 service 内部收口为一个新动作
- 如果步骤跨 service、跨聚合，统一走 orchestrator，逐步执行并记录每一步状态
- 高风险删除类动作默认不自动补偿，必须显式定义 compensation
- AI 不直接决定补偿策略，补偿策略由服务端协议声明

例如：

- “创建题库仓库 -> 批量写入题目”可以是一个编排任务
- “创建角色 -> 批量调整用户角色”也应被视为编排，而不是一个数据库事务

这样能避免把“AI 能连续下发多个动作”误写成“系统支持分布式强事务”。

#### 6.2.3 更细粒度失败分类

当前失败已经能落到执行账本里，但从企业运维角度看，还需要把“为什么失败、能否重试、谁来处理”分得更清楚。

建议把失败分类从单一 `errorCode` 扩展为至少三层：

- 失败阶段：`validation` / `policy` / `idempotency` / `execution` / `compensation` / `system`
- 失败性质：`user_fixable` / `retryable` / `manual_intervention_required`
- 失败归因：`input_field` / `target_not_found` / `dependency_conflict` / `permission_denied` / `service_exception`

建议新增字段：

- `errorStage`
- `errorClass`
- `retryable`
- `failedStepId`
- `failedAction`
- `suggestedResolution`

例如：

- JSON 字段缺失，属于 `validation + user_fixable`
- 幂等键冲突，属于 `idempotency + manual_intervention_required`
- 目标记录不存在，属于 `execution + user_fixable`
- 下游服务瞬时异常，属于 `system + retryable`

这样账本、告警、前端回填和人工介入流程才有稳定依据，避免所有失败都只显示成一条模糊错误文案。

#### 6.2.4 执行账本的筛选与可视化

这一项当前其实已经有第一版基础设施，不应再按“从零开始”理解。

截至 2026-03-31，管理后台已经支持：

- 按 `action` 过滤
- 按 `status` 过滤
- 按 `actor_id` 过滤
- 按 `created_from / created_to` 时间范围过滤
- JSON / CSV 导出
- 详情抽屉查看 request / response payload
- 对失败请求做回填重试

下一步建议补的，不是“有没有账本”，而是“账本是否足够支撑 AI 运维”：

- 增加 `batchId`、`parentExecutionId`、`stepId`、`stepIndex`
- 增加 `dryRun`、`replayed`、`retryable`、`errorStage`、`errorClass`
- 增加按动作类型、失败类别、是否可重试、是否批量任务的组合筛选
- 增加按日/周聚合的执行量、失败率、回放率统计
- 增加批量任务的树形步骤视图和时间线视图
- 增加“同一幂等键 / 同一批次 / 同一操作者”的关联跳转

推荐的可视化重点不是花哨图表，而是三种值班场景：

- 我想知道今天哪些 AI 管理动作失败了，是否需要人工处理
- 我想知道某个批量任务卡在哪一步
- 我想知道某类动作最近是否系统性出错

#### 6.2.5 动作级权限矩阵

当前管理域 AI 本质上是 `adminOnly`。这适合作为第一阶段安全边界，但对于真实企业环境仍然过粗。

下一阶段建议把权限从“是否管理员”细化到“谁可以对什么对象执行什么动作”。

建议引入动作级权限矩阵：

- 动作维度：`user.create`、`role.update`、`flow.delete`、`question_bank.repo.create` 等
- 操作者维度：超级管理员、系统管理员、题库管理员、流程管理员等
- 数据范围维度：全局、所属部门、所属题库、指定资源集合
- 风险级别维度：普通变更、高风险变更、需要二次确认的变更

建议的协议含义：

- 协议中暴露“允许动作列表”，而不是让 AI 假设自己什么都能做
- `dryRun` 可以比 `execute` 拥有更宽的权限
- 删除、密码重置、权限提升这类动作应单独标为高风险
- 某些动作即使允许执行，也应限制可修改字段

例如：

- 题库管理员可以 `question_bank.repo.create`，但不能 `role.update`
- 流程管理员可以 `flow.create` / `flow.update`，但不能 `user.password.reset`
- 普通业务管理员可以 `dryRun` 生成方案，但真正 `execute` 仍需更高权限或审批确认

只有把“动作协议”和“权限矩阵”一起暴露，AI 才能从“会写 JSON”升级为“知道自己被允许做什么”的受控助手。

目标：

把当前单动作、强受控的管理域 AI，升级为企业可用的 AI 管理助手底座。

### 6.3 第三优先级：将题库收口为统一 DSL

建议补：

- `surveyQuestion` 的明确共享类型
- 题库题标准化与校验器
- 版本号、审核状态、来源信息
- 从题库题到问卷题的稳定映射协议

目标：

让题库不只是“能装内容”，而是“能稳定驱动 AI 出题、组卷、复用和审核”。

## 7. 最终判断

如果目标是“全面接入 AI 辅助管理、出题和组卷”，当前系统已经不再是 2026-03-30 文档所描述的状态。

截至 2026-03-31，更准确的总体判断是：

- 管理域 AI：已落地第一版正式协议和执行闭环
- 问卷 JSON AI：已具备成熟的标准输入输出与 dry-run 校验基础
- 问卷逻辑 AI：已具备共享类型和后端结构校验
- 题库 AI：已进入结构化内容阶段，但仍需继续收口

所以当前最合理的推进方式是：

**以“管理域 AI 执行协议 + 问卷标准 JSON + 题库 DSL 收口”作为三条主线继续推进，而不是重新讨论是否具备 AI 接入基础。**
